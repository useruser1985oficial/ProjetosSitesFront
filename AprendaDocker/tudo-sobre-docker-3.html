    O Segredo do Cache ao Dockerizar Python e Acelerar seu Build

    Para nós irmos pra parte prática, vamos limpar todos os containers, volumes e imagens de antes.

    Vamos supor um projeto que precisa de uma versão especifica de uma biblioteca, como um projeto Python que usa o Pandas versão 1.0, e outro projeto usa outra versão do Pandas como o 2.0, e caso instalemos as duas versões de uma biblioteca no mesmo sistema operacional, o sistema pode quebrar por causa dos conflitos das bibliotecas. No entanto, com o Docker podemos contornar esse problema, onde isolaremos em um container um projeto (no caso em Python) com as bibliotecas necessárias na versão especificada.

    Primeiramente, crie um arquivo Python, como app.py, com esse código aqui:

    
from colorama import Fore, Style, init, just_fix_windows_console
import random

just_fix_windows_console()

print(Fore.BLUE + "- Gerador de Números Aleatórios - " + Style.RESET_ALL)
numero = random.randint(1, 100)

print(f"Número gerado: {Fore.GREEN}{numero}{Style.RESET_ALL}")

    A biblioteca Colorama não é uma biblioteca que vem por padrão no Python, pra a utilizarmos precisaremos instalar ela.

    Vamos criar um arquivo txt, como dependencias.txt e coloque esse código aqui:

    
colorama==0.4.6

    E faça o Dockerfile com esse código, pra criarmos a imagem:

    
FROM python:3.9

WORKDIR /app

COPY . .

RUN pip install -r dependencias.txt

CMD ["python", "app.py"]

    No caso acima, os pontos sequenciais indicam que é pra copiar tudo do diretório do nosso PC (onde está o Dockerfile) para o diretório indicado na imagem (/app).

    Para criar a imagem rodamos na linha de comando docker build -t python-color:v1 . (não esqueça do ponto) e veja se ela foi criada com docker images. Depois de criar a imagem, crie o container com ela com o comando docker run python-color:v1, se tudo der certo, ele rodará e exibirá a mensagem colorida pelo Colorama.

    No caso, usamos uma biblioteca só pra exemplificar, mas um programa que tem muitas dependências pode dar problemas e ficar pesado. Vamos exemplificar fazendo uma pequena alteração no arquivo Python e gere outra imagem com docker build -t python-color:v2 . (não esqueça do ponto), ele rodará o pip install novamente. Ele só cacheará a pasta /app, mas o pip será executado de novo, mesmo que seja a mesma versão da biblioteca. Para forçar ele a cachear o pip também, adicione uma linha acima do COPY . . indicando o arquivo de dependências assim:

    
FROM python:3.9

WORKDIR /app

COPY dependencias.txt .

RUN pip install -r dependencias.txt

COPY . .

CMD ["python", "app.py"]

    Aí, vamos deletar todos os containers e imagens e criar tudo novamente. Digite novamente docker build -t python-color:v1 . (não esqueça do ponto) pra gerar a imagem novamente. Faça uma outra alteração no código Python e gere outra imagem com docker build -t python-color:v2 . (olha o ponto). Ele gerará outra imagem, mas usará o cache das bibliotecas, que não foram alteradas. Faça o mesmo com a versão 3, usando docker build -t python-color:v3 . (o ponto).

    PS: Isso foi exemplificado com Python, mas podemos usar qualquer linguagem e suas bibliotecas.

    O Jeito Certo de Dockerizar Node.JS

    A pasta node_modules é bastante grande e pesada, e por isso, não é recomendado copiar ela do nosso computador pro container, por demorar muito, além de poder ter incompatibilidade entre versões do Node e os sistemas operacionais. O ideal é criar o Node Modules dentro do próprio container.

    Vamos supor esse código em Node (salve num arquivo index.js):

    
const EXPRESS = require("express");
const APP = EXPRESS();
const PORT = 4000;

APP.get("/", (req, res) => {
    res.json({
        canal: "Cursos",
        mensagem: "Rodando liso no Docker",
        plataforma: process.platform
    });
});

APP.listen(PORT, () => {
    console.log(`API rodando na porta ${PORT}.`);
});

    Na mesma pasta, coloque um arquivo JSON com o nome package.json, com esse código:

    
{
    "name": "api",
    "version": "1.0.0",
    "main": "index.js",
    "dependencies": {
        "express": "^5.2.1"
    },

    "scripts": {
        "start": "node index.js"
    }
}

    Aí, para instalar as dependências, rode o comando npm install na pasta onde estão os arquivos, pra ele gerar o Node Modules e podermos rodar o aplicativo. Podemos rodar o arquivo JS com npm start e ver no navegador em http://localhost:4000.

    Como podemos ver, a pasta node_modules gerada tem muitos arquivos e é bastante pesada. Nós deveremos gerar essa pasta dentro do nosso container, e não a copiar pro mesmo. Para isso criaremos um arquivo .dockerignore com esse código aqui:

    
node_modules
npm-debug.log
Dockerfile
.git

    E no Dockerfile, colocaremos isso:

    
FROM node:20-alpine

WORKDIR /app

COPY package*.json ./

RUN npm install

COPY . .

EXPOSE 4000

CMD ["npm", "start"]

    E pra criar a imagem basta rodarmos o comando docker build -t api-node:v1 . (não esqueça do ponto). Depois rode docker images pra ver se a imagem foi criada. E depois crie o container no modo detach com docker run -d -p 5000:4000 --name minha-api api-node:v1. Para ver se o JSON está rodando na porta especificada, acesse no navegador http://localhost:5000.

    Projeto Final

    Vamos fazer um projeto com Node e MySQL, para isso, crie uma pasta com o nome ProjetoFinal e dentro dela, uma pasta com o nome api, onde criaremos nossos arquivos. Na raiz do ProjetoFinal, crie um arquivo docker-compose.yaml (ou yml) e o arquivo .env. Dentro da pasta api, crie um arquivo package.json.

    No arquivo package.json, vamos colocar o que instalaremos no nosso projeto, com esse código:

    
{
    "name": "todo",
      "dependencies": {
        "express": "^5.2.1",
        "mysql2": "^3.6.0"
    },
    "scripts": {
        "start": "node index.js"
    }
}

    Ainda dentro de API, crie o arquivo index.js, com esse código:

    
const EXPRESS = require("express");
const MYSQL = require("mysql2");
const APP = EXPRESS();
const PORT = 4000;
let pool;

// Função para criar o pool
function createPool() {
    pool = MYSQL.createPool({
        host: process.env.DB_HOST,
        user: process.env.DB_USER,
        password: process.env.DB_PASSWORD,
        database: process.env.DB_NAME,
        waitForConnections: true,
        connectionLimit: 10,
        queueLimit: 0
    });
}

// Função de retry
function connectWithRetry() {
    createPool();

    pool.getConnection((error, connection) => {
        if(error) {
            console.log("Banco não disponível, tentando novamente em 5s...");
            setTimeout(connectWithRetry, 5000);
        }
        else {
            console.log("Conectado ao MySQL com sucesso!");
            connection.release();
        }
    });
}

// inicia tentativa de conexão
connectWithRetry();

APP.get("/tarefas", (req, res) => {
    if(!pool) {
        return res.status(500).send("Banco ainda não inicializado!");
    }

    pool.query("SELECT * FROM tarefas", (err, results) => {
        if(err) {
            console.error(err);
            return res.status(500).send("Erro no banco: " + err.message);
        }
        res.json(results);
    });
});

APP.listen(PORT, () => {
    console.log(`API rodando na porta ${PORT}.`);
});

    Ainda dentro de api, crie o Dockerfile com esse código:

    
FROM node:20-alpine

WORKDIR /app

COPY package*.json ./

RUN npm install

COPY . .

EXPOSE 4000

CMD ["npm", "start"]

    Voltando à raiz do ProjetoFinal, em .env, coloque isso:

    
DB_HOST=meubanco
MYSQL_ROOT_PASSWORD=suasenhaforte
MYSQL_DATABASE=listatarefas
MYSQL_USER=usuario
MYSQL_PASSWORD=suasenhaforte

    PS: Os nomes usados no .env não são os mesmos no index.js, mas na nossa imagem vamos criar outro arquivo de variáveis de ambiente, e ele utilizará as variáveis do Docker Compose.

    Coloque isso no Docker Compose, na raiz de ProjetoFinal:

    
services:
  minha-api:
    build: ./api # O Docker vai entrar na pasta e procurar o Dockerfile
    ports:
      - "5000:4000"
    environment:
      - DB_HOST=meubanco
      - DB_USER=root
      - DB_PASSWORD=${MYSQL_ROOT_PASSWORD}
      - DB_NAME=${MYSQL_DATABASE}
    depends_on:
      - meubanco
    restart: always # Se o banco demorar e a API cair, ela tenta de novo

  meubanco:
    image: mysql:5.7
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_DATABASE: ${MYSQL_DATABASE}
    volumes:
      - dados-do-banco:/var/lib/mysql
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql

  adminer:
    image: adminer
    ports:
      - "8080:8080"

volumes:
  dados-do-banco:

    No código acima, ele criará três containers para o nosso projeto, além de um volume para armazenamento persistente.

    Na raiz do projeto, crie o arquivo init.sql com esse código:

    
use listatarefas;

create table tarefas (
    id int auto_increment primary key,
    descricao varchar(50) not null
)
default charset = utf8;

insert into tarefas (descricao) values
('Se inscrever no canal'),
('Estudar Docker'),
('Dar like no vídeo'),
('Seguir no Insta');

    No Prompt, na pasta da raiz do projeto, digite o comando docker compose up -d --build. Ele executará em segundo plano, e pra ver os logs em tempo real digite docker compose logs -f. Rode no navegador do computador o link http://localhost:5000/, mas note que ele dará erro (exibindo Cannot GET /), mas se acessar http://localhost:5000/tarefas ele retornará os dados do banco em JSON (ele demora um pouco pra subir o MySQL).

    Como nós fizemos um volume para persistir dados, podemos excluir os containers, pois tudo estará salvo no volume, podemos rodar docker compose down para ele excluir os containers. Aí para vermos a criação dos container novamente usamos o comando docker compose up -d sem o build, pois não precisará criar o volume de novo.

    Com isso, nós criamos nossos containers separados para cada coisa no nosso projeto.
